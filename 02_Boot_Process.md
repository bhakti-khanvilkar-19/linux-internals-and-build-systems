# Detailed Boot Process: x86 PC vs. Embedded Systems

This document provides a granular, step-by-step breakdown of the boot sequence for two different architectures:
1.  **Standard x86 PC**: The kind of machine that runs desktop Linux (like Ubuntu), Windows, etc.
2.  **Embedded Processor**: Common in custom hardware, IoT devices, automotive systems, etc. (e.g., ARM Cortex-A, RISC-V).

### ğŸŸ¢ STAGE 0 â€” You press the power button (physical reality)

Power button connects a signal to the motherboard.

The **Power Supply Unit (PSU)** (the box that converts wall electricity into stable voltages for the computer) starts delivering power rails:
- `+12` volts
- `+5` volts
- `+3.3` volts

Once voltages stabilize, the power supply sends a signal called the **Power Good signal**.
- **Meaning**: â€œElectricity is stable. Safe to start.â€
- Only after this does the processor begin executing instructions.

### ğŸŸ¢ STAGE 1 â€” CPU Reset Vector (The True Beginning)

When the processor starts, it does NOT search for an operating system.

Processors are dumb and obedient. They simply jump to a hard-coded memory address called the **Reset Vector**.
- **Reset Vector** = A predefined memory location where the CPU starts executing instructions after power-on.
- For modern x86 computers, this address is `0xFFFFFFF0`.

This address is mapped to firmware stored on the motherboard. The processor executes whatever instruction lives there.

> **How does the CPU know where to start?**
> It doesnâ€™t â€œknowâ€. Its hardware wiring forces it to start at that fixed address.

### ğŸŸ¢ STAGE 2 â€” Firmware (BIOS or UEFI)

The reset vector points into motherboard firmware stored in **Flash memory** (a rewritable non-volatile chip on the motherboard).

This firmware is either:
- **BIOS** (Basic Input Output System â€” older style)
- **UEFI** (Unified Extensible Firmware Interface â€” modern replacement)

This firmware is written by motherboard manufacturers. *Updating your BIOS* means rewriting that flash chip.

**What firmware does first (very early boot):**
1.  **CPU Initialization**: Sets operating mode, configures registers, and prepares the memory controller.
2.  **POST (Power-On Self-Test)**: The firmware checks critical hardware:
    - RAM works
    - Keyboard present
    - Processor functional
    - Graphics initialized
    - Storage detectable
    *If RAM fails, the system typically beeps or refuses to boot.*
3.  **Hardware Discovery**: Firmware scans buses like PCI Express, SATA/NVMe, and USB to build a list of bootable devices.
4.  **Boot Device Selection**: The firmware reads a pre-configured boot order (e.g., 1. SSD, 2. USB Drive, 3. Network) and tries each until it finds a bootable one.

### ğŸŸ¢ STAGE 3 â€” Finding the Bootloader

This depends on the firmware type.

#### OLD BIOS METHOD
- BIOS reads the **Master Boot Record (MBR)** â€” the first 512 bytes of the disk.
- This tiny sector contains the partition table and a small piece of bootloader code.
- BIOS loads this sector into RAM and jumps to it. That tiny code then loads the *real* bootloader (like GRUB).

#### MODERN UEFI METHOD (Your Ubuntu likely uses this)
- UEFI reads a special partition called the **EFI System Partition (ESP)**, which is a small FAT filesystem.
- Inside, it looks for a bootloader executable, for example: `/EFI/ubuntu/grubx64.efi`.
- UEFI loads and runs this file directly. No 512-byte hacks are needed.

### ğŸŸ¢ STAGE 4 â€” Bootloader (GRUB)

**GRUB (GRand Unified Bootloader)** is the program responsible for loading the operating system.

GRUB now:
1.  Reads its configuration file.
2.  Shows an OS menu (if configured) and waits for selection.
3.  Loads two critical files into RAM:
    - **Kernel**: A file like `/boot/vmlinuz-6.x`
    - **initramfs**: A file like `/boot/initrd.img`

> **What is initramfs?**
> **Initial RAM Filesystem**. It's a compressed archive unpacked into RAM, containing a tiny Linux filesystem with drivers, scripts, and BusyBox utilities. Think of it as an *emergency starter OS* used before the real one is accessible.

GRUB then passes control to the kernel and jumps to its entry point. The bootloaderâ€™s job is now done.

### ğŸŸ¢ STAGE 5 â€” Kernel Begins Executing

The kernel first:
1.  **Decompresses itself**: The kernel is stored compressed to save disk space, so it expands itself into RAM.
2.  **Sets up core systems**: It initializes memory management (page tables, virtual memory) and interrupt handlers. Without this, multitasking is impossible.
3.  **Initializes hardware drivers**: The kernel detects processor cores, RAM size, timers, storage controllers, USB buses, and basic graphics (framebuffer).
4.  **Mounts initramfs as temporary root**: At this moment, the root filesystem `/` is the one in RAM, not your real disk.

### ğŸŸ¢ STAGE 6 â€” Running `init` inside initramfs

The kernel executes the `/init` script inside the initramfs. This script:
1.  Loads necessary storage drivers (e.g., for NVMe or SATA).
2.  Loads filesystem modules (e.g., `ext4`, `btrfs`).
3.  Handles disk encryption unlock if needed.
4.  Activates LVM (Logical Volume Manager) if used.
5.  Finds the real root partition (e.g., `/dev/nvme0n1p2`).
6.  Performs a **`switch_root`**: This crucial step replaces the temporary RAM filesystem with the real disk filesystem.

Now, `/` points to your real Ubuntu filesystem, and the initramfs is no longer in active use.

### ğŸŸ¢ STAGE 7 â€” Starting the Real `init` System

The kernel now runs `/sbin/init` from the real root filesystem. In modern Ubuntu, this is **`systemd`**.

`systemd` becomes **PID 1** (Process Identifier number 1) â€” the first userspace process. If PID 1 dies, the system crashes.

### ğŸŸ¢ STAGE 8 â€” `systemd` Builds the Running System

`systemd` is a powerful service manager that brings the system to a usable state. It:
- Mounts all filesystems listed in `/etc/fstab`.
- Starts the device manager (`udevd`).
- Starts networking, logging, and time synchronization services.
- Starts login services.

For a graphical system, it launches a **display manager** (the login screen). For a terminal-only system, it launches **`getty`** (the program that shows a login prompt on a terminal).

### ğŸŸ¢ STAGE 9 â€” Login

You authenticate. `systemd` then starts your shell session or desktop environment.

**Boot complete.** You are now inside Linux userspace.

### ğŸ”µ STAGE 1 â€” Power-On Reset & Boot ROM

1.  **Power-On Reset (POR)**: When power is applied, the System on a Chip (SoC) is held in a reset state until power is stable.
2.  **Boot ROM**: The processor begins execution at a hard-coded **reset vector**, just like a PC. This vector points to a small, permanent, on-chip **Boot ROM**. This code is burned into the silicon by the chip manufacturer and cannot be changed.

The Boot ROM's only job is to find the *next-stage bootloader* from a bootable source.

### ğŸ”µ STAGE 2 â€” Finding the First-Stage Bootloader (FSBL)

The Boot ROM checks a sequence of boot devices, determined by hardware configuration (e.g., fuses burned at the factory or GPIO pin states). A common sequence is:
1.  eMMC flash memory
2.  SD Card
3.  QSPI flash memory
4.  USB (in a special device firmware upgrade mode)
5.  UART (waiting for a bootloader image to be sent over serial)

The Boot ROM looks for a valid **First-Stage Bootloader (FSBL)** or **Secondary Program Loader (SPL)** at a specific offset on one of these devices. Once found, it loads it into the SoC's internal SRAM (a small, fast, on-chip RAM).

### ğŸ”µ STAGE 3 â€” First-Stage Bootloader (FSBL / SPL)

The FSBL (often a part of a larger bootloader like U-Boot, called SPL) is a very small program with a few key jobs:
1.  **Initialize DRAM**: The main system memory (DDR4, etc.) is not yet usable. The FSBL configures the memory controller to bring the DRAM online.
2.  **Load the Second-Stage Bootloader**: It copies the main, full-featured bootloader (like `u-boot.img`) from the boot media (e.g., eMMC, SD card) into the now-usable DRAM.
3.  **Jump to the Second-Stage Bootloader**.

### ğŸ”µ STAGE 4 â€” Second-Stage Bootloader (U-Boot)

**U-Boot (Das U-Boot - the Universal Bootloader)** is the de facto standard for embedded Linux. It is a powerful, configurable bootloader that runs from DRAM.

Its responsibilities include:
1.  **Initialize Peripherals**: Sets up essential hardware needed for booting, such as the console (UART), storage controllers (eMMC/SD), and network interfaces (Ethernet).
2.  **Provide a Console Interface**: U-Boot typically provides a command-line shell over a serial (UART) connection, allowing for debugging, configuration, and manual boot commands.
3.  **Load Kernel and Device Tree**: It loads two key files from storage into DRAM:
    - **Kernel Image**: e.g., `uImage` or `zImage`.
    - **Device Tree Blob (DTB)**: A file like `soc_name.dtb`. The DTB is a data structure that describes the hardware present on the board (which GPIOs are used for what, which memory-mapped devices exist, etc.). This allows the same Linux kernel to be used on different boards with different hardware layouts.
4.  **Modify Device Tree (Optional)**: U-Boot can modify the DTB in memory before passing it to the kernel, for example, to add the kernel command line.
5.  **Jump to the Kernel**: U-Boot passes control to the Linux kernel, providing the memory addresses of the kernel and the DTB.

### ğŸ”µ STAGE 5 â€” Kernel and `init`

From here, the process is similar to the PC boot, but simpler:
1.  **Kernel Decompression & Initialization**: The kernel decompresses, sets up memory management, and uses the **Device Tree** to know which drivers to initialize for the specific hardware on the board.
2.  **Mount Root Filesystem**: The kernel mounts the root filesystem, as specified in the kernel command line (e.g., `root=/dev/mmcblk0p2`). There is often **no initramfs** in simpler embedded systems, as all necessary drivers are built into the kernel.
3.  **Execute `init`**: The kernel runs `/sbin/init`. In embedded systems, this can be `systemd`, but is often a much simpler init system like **BusyBox `init`** or **SysVinit** to save space and reduce complexity.
4.  **Launch Application**: The init system starts the necessary services and, finally, launches the main embedded application that the device was built to run (e.g., a router's web interface, a car's infotainment UI).

The user typically never sees a login prompt; the device boots directly into its designated function.

## Part 2B: Secure Boot in Embedded Systems

> **Diagram available**: Open [diagrams/embedded_boot_cycle.drawio](diagrams/embedded_boot_cycle.drawio) in VS Code or draw.io for an interactive visual of the secure boot chain.

Secure boot ensures that only **authorized, unmodified software** runs on your device. This is critical for:
- **IoT devices**: Prevent malware from taking over smart home devices
- **Automotive**: Ensure ECU firmware hasn't been tampered with
- **Medical devices**: Regulatory requirement for safety
- **Payment terminals**: PCI compliance requires secure boot

The **hardware anchor** that starts the chain. It cannot be modified by software.

#### dm-verity (Root Filesystem Verification)
Extends trust to the entire root filesystem.

```bash
# dm-verity creates a hash tree of the filesystem
# Kernel verifies each block as it's read

# Setup (at build time):
veritysetup format /dev/mmcblk0p2 /dev/mmcblk0p3   # Creates hash partition

# Boot (kernel cmdline):
root=/dev/dm-0 dm-mod.create="vroot,,,ro,0 SIZE verity 1 DATA_DEV HASH_DEV 4096 4096 BLOCKS HASH_START sha256 ROOT_HASH SALT"
```

#### IMA/EVM (Individual File Verification)
Measures and verifies each executable/config file.

```bash
# IMA = Integrity Measurement Architecture
# EVM = Extended Verification Module

# Files have extended attributes with signatures:
getfattr -m - -d /usr/bin/bash
# security.ima = 0x030204...  (signature)
# security.evm = 0x050204...  (signature of all xattrs)
```

### ğŸš¨ What Happens When Verification Fails?

| Failure Point | Typical Behavior |
|---------------|------------------|
| **Boot ROM** | Device hangs, enters recovery mode, or loops |
| **SPL** | Halts, may print error to UART |
| **U-Boot** | Refuses to boot kernel, drops to U-Boot shell |
| **Kernel** | Panic, or falls back to recovery partition |
| **dm-verity** | Kernel panic or read-only mode (depends on config) |

## Part 3: Detailed Comparison Tables

### Boot Stage Comparison

| Stage | x86 PC (UEFI) | Embedded (ARM/RISC-V) |
|-------|---------------|------------------------|
| **Power-On** | PSU stabilizes, sends "Power Good" | POR circuit holds reset until stable |
| **First Code** | UEFI firmware in Flash chip | Boot ROM burned into silicon |
| **Firmware Source** | Motherboard manufacturer (updatable) | Chip manufacturer (permanent) |
| **DRAM Init** | Done by UEFI firmware | Done by FSBL/SPL |
| **Bootloader** | GRUB (single stage) | SPL â†’ U-Boot (two stages) |
| **Hardware Description** | ACPI tables (from firmware) | Device Tree Blob (from storage) |
| **Kernel Loading** | GRUB loads vmlinuz + initramfs | U-Boot loads zImage + DTB |
| **Early Rootfs** | initramfs (in RAM) | Often none (drivers built-in) |
| **Init System** | systemd (full featured) | BusyBox init (minimal) |
| **End Result** | Login screen / Desktop | Application auto-starts |

### Key Component Comparison

| Component | x86 PC | Embedded | Purpose |
|-----------|--------|----------|---------|
| **Reset Vector** | `0xFFFFFFF0` | Chip-specific (e.g., `0x00000000`) | Where CPU starts executing |
| **Firmware Storage** | SPI Flash (8-32MB) | On-chip ROM + external Flash | Holds first-stage code |
| **DRAM** | DIMM slots (removable) | Soldered chips (fixed) | Main system memory |
| **Boot Media** | NVMe/SATA SSD | eMMC/SD/QSPI Flash | Where OS is stored |
| **Console** | VGA/HDMI + keyboard | UART serial port | Debug/interaction interface |
| **Bootloader Config** | `/boot/grub/grub.cfg` | U-Boot environment variables | Boot parameters |

## Part 5: Practical Commands to Inspect Boot

### On Ubuntu (x86 PC)

```bash
# View boot messages (kernel ring buffer)
dmesg | head -100

# View systemd boot timeline
systemd-analyze

# View detailed boot blame (which services took longest)
systemd-analyze blame

# See boot sequence as a graph (generates SVG)
systemd-analyze plot > boot.svg

# View kernel command line (parameters passed by GRUB)
cat /proc/cmdline

# List installed kernels
ls -la /boot/vmlinuz*

# View UEFI boot entries
efibootmgr -v

# Check if booted via UEFI or BIOS
[ -d /sys/firmware/efi ] && echo "UEFI" || echo "BIOS"
```

### On Embedded Linux (via serial console)

```bash
# In U-Boot (before kernel boots):
printenv              # Show all U-Boot environment variables
bdinfo                # Show board info (DRAM size, etc.)
mmc info              # Show eMMC/SD card info

# In Linux:
dmesg | head -100     # View kernel boot messages
cat /proc/device-tree/model  # Show board name from Device Tree
ls /proc/device-tree/ # Explore Device Tree nodes

# View kernel command line
cat /proc/cmdline
```
